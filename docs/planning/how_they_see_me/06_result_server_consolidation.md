# 프론트엔드 결과 템플릿 제거 및 서버 단일 소스 구조 전환 명세

이 문서는 “결과 텍스트 템플릿을 백엔드 단일 소스로 운영”하기 위한 프론트엔드 정리 작업 명세입니다.
목표는 **프론트에서 결과 문장/템플릿을 완전히 제거**하고, **서버 응답만으로 결과 화면을 렌더링**하도록 구조를 정리하는 것입니다.

---

# 1. 변경 목적

현재 상태
- 결과 텍스트 템플릿이 백엔드 서버에 존재함
- 동시에 프론트 프로젝트에도 템플릿이 남아있음 (중복 상태)

이 상태는 장기적으로 아래 문제를 유발합니다.

문제점
- 서버 문구 수정 시 프론트와 불일치 발생
- 프론트 재배포 없이는 문구 수정 불가
- A/B 테스트 불가능
- 콘텐츠 운영 난이도 증가

따라서 결과 텍스트는 **콘텐츠 데이터**로 간주하고
**백엔드를 단일 소스(Single Source of Truth)** 로 확정합니다.

---

# 2. 최종 아키텍처

## 프론트엔드 역할
- 결과 텍스트 생성 ❌
- 결과 템플릿 보관 ❌
- 키워드 조합 ❌

프론트는 오직 아래만 수행합니다.

프론트 책임 범위
1) 서버에서 결과 데이터 요청
2) 받은 데이터를 화면에 렌더링
3) 로딩 / 에러 처리

즉, 프론트는 **뷰 렌더러** 역할만 수행합니다.

---

## 백엔드 역할

백엔드 책임 범위
- 결과 템플릿 100% 보관
- 키워드 → 문장 조합 로직
- 결과 문장 생성
- 결과 API 제공

결과 문장은 모두 서버에서 완성된 상태로 내려옵니다.

---

# 3. 프론트에서 반드시 삭제해야 할 항목

프로젝트 전역에서 아래 요소를 전부 제거합니다.

삭제 대상
- 결과 텍스트 템플릿 파일
- 키워드 → 문장 조합 함수
- 결과 생성 유틸 함수
- 결과 더미 데이터(JSON)
- 결과 문장 상수 파일

대표적인 삭제 키워드 예시
- resultTemplates
- resultGenerator
- johariTemplates
- buildResultText
- resultDummyData

중요:
프론트 코드 어디에도 “결과 문장 생성 로직”이 남아있으면 안 됩니다.

---

# 4. 프론트에 남겨야 하는 것 (유지 항목)

삭제하면 안 되는 항목도 명확히 정의합니다.

유지 대상

## 1) 타입 정의
서버 응답 타입은 반드시 유지합니다.

예시 타입
ResultResponse {
openSelf: {
title: string
subtitle: string
description: string
keywords: string[]
}
blindSelf: { ... }
hiddenSelf: { ... }
unknownSelf: { ... }
}

yaml
코드 복사

프론트는 이 타입 기반으로만 렌더링합니다.

---

## 2) 로딩 상태 UI
결과 요청 중 표시
- Skeleton UI
- 로딩 텍스트

---

## 3) 에러 처리 UI
API 실패 시 표시

예시 문구
- 결과를 불러오지 못했습니다.
- 잠시 후 다시 시도해주세요.

---

# 5. 결과 페이지 데이터 흐름

## API 호출 흐름

페이지 진입
/result/{shareId}

yaml
코드 복사

프론트 동작 순서

1) shareId 추출
2) 결과 API 호출
3) 응답 수신 후 렌더링

---

## 서버 응답 구조 (예시)

{
"openSelf": {
"title": "개방된 자아",
"subtitle": "나도 알고, 남도 아는 나",
"description": "사람들이 당신을 떠올릴 때...",
"keywords": ["친절한","신뢰할 수 있는"]
}
}

yaml
코드 복사

중요 규칙
- description은 이미 완성된 문장
- 프론트에서 절대 가공하지 않음

---

# 6. 프론트 렌더링 규칙

렌더링 순서 (영역별 동일)

각 영역 표시 구성
1) 영역 타이틀
2) 서브타이틀
3) 설명 텍스트
4) 키워드 뱃지 목록

프론트는 단순 출력만 수행합니다.

금지 사항
- 키워드로 문장 생성 ❌
- 문장 수정 ❌
- 단어 교체 ❌

---

# 7. 운영 관점 이점

이 구조 전환으로 얻는 효과

운영 개선
- 문구 수정 시 프론트 배포 불필요
- A/B 테스트 가능
- 감정 톤 지속 개선 가능
- 콘텐츠 업데이트 속도 향상

즉, 결과 텍스트는 “코드”가 아니라
**운영 가능한 콘텐츠 데이터**로 전환됩니다.

---

# 8. 최종 결론

프론트 역할
- 결과 템플릿 완전 제거
- 서버 응답만 렌더링

백엔드 역할
- 결과 문장 생성 100% 담당

핵심 원칙
결과 텍스트는 서버의 책임이다.
프론트는 보여주기만 한다.
