# 03. 치코런 문제 템플릿 생성 및 관리 명세

## 1. 목표
- 레벨(1~100)마다 해당 레벨에 정확히 맞는 난이도의 문제를 생성.
- 문제가 순서대로 출제되며, 유형을 최대한 다양하게 구성.
- 모든 문제는 정답이 명확히 하나이며, 오류(구멍, 모호한 답, 오답 해설 오류 등)가 절대 발생하지 않도록 철저한 검증 규칙을 적용.

## 2. 문제 출제 방식
### 순차 출제 (Random X)
- 각 레벨별로 문제는 고정된 순서로 출제됨.
- 학생이 재접속하거나 레벨을 이동해도 `progressIndex + seed`에 따라 이전 문제 다음부터 정확히 이어서 출제됨.

### 레벨별 문제 수
- **초급 (1~30)**: 최소 12문제
- **중급 (31~70)**: 최소 15문제
- **고급 (71~100)**: 최소 18문제
- 한 번 클리어한 레벨은 순환하지 않고 다음 레벨로 이동 (복습 모드 별도 제공 예정).

## 3. 문제 유형 다양화
문제는 아래 유형을 레벨 구간별로 균형 있게 배분한다. 같은 유형이 연속으로 2번 나오지 않도록 순서를 설계.

### 초급 (1~30)
- `vocab` (단어 뜻)
- `basic_grammar` (시제, 전치사, 관계대명사)
- `simple_fact` (지문 속 사실 찾기)
- `synonym` (동의어)

### 중급 (31~70)
- `context_vocab` (문맥 속 어휘)
- `inference` (추론)
- `main_idea` (주제 파악)
- `purpose` (목적)
- `connector` (연결어/논리)
- `summary` (짧은 요약)

### 고급 (71~100) – 수능 유형 중심
- `long_inference` (장문 추론)
- `attitude` (필자 태도)
- `blank_grammar` (빈칸 문법+어휘)
- `order_arrange` (순서 배열)
- `comparison` (두 지문 비교)
- `true_false` (내용 일치/불일치)
- `integrated` (통합형: 지문 2개)
- `purpose_detailed` (세부 목적)

각 레벨당 유형 분포는 템플릿 생성 시 자동 균형 맞춰서 생성.

## 4. 문제 데이터 구조
```typescript
interface Problem {
  id: string;                    // UUID 또는 level-problemIndex
  level: number;                 // 1~100
  difficulty: 'easy' | 'medium' | 'hard';
  orderIndex: number;            // 같은 레벨 내 순서 (1부터 시작)
  passage: string;               // 지문 (고급은 200~450단어)
  question: string;              // 문제 stem
  choices: [string, string, string, string];  // 정확히 4개
  correctAnswerIndex: number;    // 0, 1, 2, 3 중 반드시 하나
  explanation: string;           // 오답 시에만 사용 (2~3줄 MZ 말투)
  questionType: string;          // 상단에 나열한 유형 중 하나
  wordCount: number;
  createdAt: string;
}
```

## 5. 저장 방식
- 모든 문제를 미리 생성해서 서버 DB(또는 JSON 파일)에 저장.
- **이유**:
    - 용량 충분 (1,500문제 정도, 지문 포함 최대 5~8MB 수준).
    - 순차 출제를 위해 미리 만들어 놓는 것이 가장 안전하고 정확함.
    - 템플릿만 만들어 그때그때 생성하면 순서 보장 어려움, 난이도 편차, 오류 발생 위험 증가.
    - 나중에 문제 수정/보완 용이.

### 구현 방법
- 백엔드에 `problemSeeder.ts` (또는 Python 스크립트)를 작성하여 한 번에 모든 문제 생성.
- MySQL/PostgreSQL 또는 JSON 파일로 저장. (현재 프로젝트는 MySQL/Prisma 사용 중으로 보임)
- 프론트엔드는 API `GET /problems/level/{level}/index/{progressIndex}` 로 순서대로 받아옴.

## 6. 문제 생성 검증 규칙 (구멍 방지)
아래 조건을 자동 체크하며, 하나라도 위반되면 해당 문제를 재생성한다.
- 정답은 반드시 하나이며, 다른 선택지가 정답이 될 수 없음.
- 정답 선택지가 지문에 직접적·명확하게 근거가 있어야 함. (추측/모호 금지)
- 오답 선택지는 지문과 완전히 반대되거나, 지문에 없는 내용이거나, 부분적으로만 맞는 함정으로 구성.
- 빈칸 문제의 경우 정답이 문법·어휘상 유일하게 자연스러운 것.
- 수능형 문제는 실제 수능 해설 스타일로 검증.
- `explanation`은 오답일 때만 사용되며, 정답 시에는 절대 노출되지 않음.
- 지문과 문제 stem에 모순이 없음.
- 선택지 4개는 길이와 난이도가 비슷하게 구성.
- `explanation`은 MZ 말투 + 직관적 + 2~3줄.

## 7. 오답 해설 규칙 (MZ 스타일)
- 최대 2~3줄 (한 줄당 40자 이하 권장).
- 말투: “야”, “진짜”, “이 부분이 핵심이야”, “헷갈리기 딱 좋지?”, “이렇게 생각하면 바로 잡혀!” 등 친근+직설.
- 예시: “야 이건 지문에서 ‘however’ 때문에 완전 반대 의미야! 여기서 답은 2번이야 🔥”
- 정답인 경우 해설 없이 바로 다음 문제로 이동 (기존 UX 유지).

## 8. 구현 우선순위
1. `problemSeeder` 스크립트 작성 (레벨별 난이도·유형 자동 배분)
2. Mock 데이터 1~10레벨까지 먼저 생성해서 프론트 테스트
3. 백엔드 API 연동 (`/problems` 엔드포인트)
4. 프론트에서 `progressIndex` 기반 순차 로드

## AI 작업 지침

### 목적
- 고품질의 영어 문제를 레벨별로 자동 생성 및 관리함.
- 문제 출제의 정확성, 난이도 조절, 오답 해설의 스타일(MZ형)을 일관되게 유지함.

### 작업 단계
1. **정적 데이터 생성**: `problemSeeder`를 통해 레벨별, 유형별 최소 문제 수를 준수하여 생성. (총 ~1,500문제)
2. **검증**: 생성된 문제가 상기 명시된 6번 검증 규칙을 준수하는지 LLM(OpenAI/Claude 등)을 활용해 자가 교정 및 필터링.
3. **DB 저장**: `Prisma` 또는 `Sequelize` (기존 DB 라이브러리에 맞춤)를 사용하여 생성된 문제를 DB에 삽입.
4. **백엔드 구현**: `ProblemService` 및 `Controller`를 통해 `/api/problems/level/:level/index/:index` 형태의 API를 노출.
5. **프론트엔드 연동**: `ProblemContext` 또는 전역 상태를 활용해 학습 시 순차적으로 문제를 가져와 렌더링.

### 주의사항
- **정답 중복 금지**: 한 선택지에 정답이 두 개가 될 수 있는 모호한 케이스 철저히 배제.
- **MZ 말투 유지**: `explanation`은 반드시 친근하고 직설적인 말투(`반말+존댓말 섞인 형태`)로 작성.
- **연속 유형 배제**: 같은 문제 유형이 연속 2회 이상 나오지 않도록 순서(orderIndex) 조정 필수.
- **지문 길이**: 고급 레벨 지문은 반드시 최소 200단어 이상(수능 단문 길이)을 보장.
