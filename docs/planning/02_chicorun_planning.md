# 치코런(Chicorun)

## 서비스 기획

## 1. 서비스 개요

치코런 영어 학습 서비스는
교사가 클래스를 생성하고 학생들이 참여하여 **게임 방식으로 영어 학습을 진행하는 웹 서비스**입니다.
학생들은 닉네임과 비밀번호만으로 빠르게 접속하여 학습을 시작할 수 있으며
👉 **최초 로그인 이후에는 JWT 기반 인증으로 유지됩니다**
학습 진행 상황은 데이터베이스에 저장됩니다.
교사는 학생들의 학습 진행 상황을 확인할 수 있습니다.

---

## 2. 서비스 목표

- 교사가 수업에서 쉽게 활용 가능한 영어 학습 서비스 제공
- 학생들이 게임처럼 레벨을 클리어하며 영어 학습
- 로그인 절차 최소화
- 서버 부하 최소화
- 데이터 구조 단순화
- 초등 ~ 고등 단계별 어휘 학습 제공

---

## 3. 사용자 구조

### 👨🏫 교사

- 카카오 로그인 사용(하루상자에 이미 구현되어있음, 구현할 필요 없고 연동만)
- 클래스 생성 가능
- 학생 관리 가능
- 학습 진행 상황 확인 가능

### 👨🎓 학생

- 클래스 링크로 접속 (직접 코드 입력 대신 링크를 통한 입장)
- 닉네임 + 비밀번호 등록
- **학생은 하나의 클래스에만 속할 수 있음**
- 닉네임 2~10글자(한글/영문/숫자), 비밀번호 4~8글자(숫자만)
- 학습 진행

👉 **인증 방식**

- 최초 로그인 시 (닉네임 + 비밀번호 검증)
- 이후 **JWT 발급 및 사용 (Stateless 인증)**

```
JWT payload
req.user = {
  studentId,
  classCode
}
```

## 7. 🎯 레벨 결정 및 시작 로직 (Option 1)

### 7.1 목표
레벨 테스트 기능이 없으므로, 학생이 처음 학습을 시작할 때 적절한 레벨에서 시작할 수 있도록 하고, 빠르게 자신의 실력에 맞는 레벨로 조정될 수 있게 한다.

### 7.2 전체 흐름
1. 학생이 로그인 후 학습 메인 페이지에 진입
2. 시작 레벨 선택 모달 자동으로 표시
3. 학생이 자신의 대략적인 실력을 선택
4. 선택된 범위의 중간 레벨을 `currentLevel`로 설정
5. 학습 시작 후 5~10문제 정도 풀면서 자동 레벨 조정 수행
6. 이후에는 언제든지 상단에서 레벨 변경 가능

### 7.3 시작 레벨 선택 모달 상세
- **제목**: “시작 레벨을 선택해주세요”
- **설명**: “현재 영어 실력에 맞는 레벨에서 시작하면 더 재미있게 공부할 수 있어요!”
- **선택 옵션**:
    - **완전 초보예요** → 추천 레벨: 1 ~ 15 → 실제 시작 레벨: **8레벨**
    - **기초는 좀 알아요** → 추천 레벨: 11 ~ 35 → 실제 시작 레벨: **23레벨**
    - **중급 정도예요** → 추천 레벨: 31 ~ 60 → 실제 시작 레벨: **45레벨**
    - **꽤 자신 있어요** → 추천 레벨: 56 ~ 85 → 실제 시작 레벨: **70레벨**
    - **고급 수준이에요** → 추천 레벨: 81 ~ 100 → 실제 시작 레벨: **90레벨**
- **UI 요구사항**:
    - 각 옵션은 카드 형태로 시각적으로 구분
    - 선택 시 바로 “시작하기” 버튼 활성화
    - “나중에 선택할게요” 버튼 제공 → 선택 시 기본값으로 1레벨 시작

### 7.4 자동 레벨 조정 로직 (학습 중 적용)
학습 시작 후 첫 10문제 동안 아래 규칙으로 `currentLevel` 자동 조정:
- **연속 정답 3회 이상** → 레벨 +1
- **연속 정답 5회 이상** → 레벨 +2
- **연속 정답 8회 이상** → 레벨 +3
- **오답 2회 이상 (연속 또는 누적)** → 레벨 -1
- **오답 4회 이상** → 레벨 -2
- **조정 제한**: 최대 ±5레벨까지만 적용 (선택한 시작 레벨 기준)
- **종료 조건**: 10문제 이후에는 자동 조정 종료
- **저장**: 조정된 레벨은 즉시 UI에 반영 + 로컬/서버에 저장

### 7.5 추가 기능
- 학습 화면 상단에 현재 레벨 표시 (예: "현재 45레벨")
- 옆에 작은 “레벨 변경” 버튼 추가
- “레벨 변경” 버튼 클릭 시 → 시작 레벨 선택 모달 다시 표시
- 모달에서 다시 선택하면 `currentLevel` 즉시 변경 및 학습 문제 재로드

### 7.6 상태 관리 (Data Schema)
- `student.currentLevel` (number): 현재 학습 중인 레벨 (1~100)
- `student.startLevel` (number): 최초 선택한 레벨 (기록용)
- `adjustmentCount` (number): 자동 조정한 문제 수 (10까지 카운트)

### 7.7 주의사항
- 처음 모달은 학습 시작 전에 반드시 한 번만 표시 (localStorage로 관리)
- 이미 `currentLevel`이 저장되어 있는 학생(재접속)은 모달을 띄우지 않고 바로 학습 시작
- 레벨과 `progressIndex`의 관계: 1레벨 = 100문제 단위로 매핑 (예: 1레벨 = index 0~99, 45레벨 = index 4400~4499)

## 8. 🧠 문제 생성 시스템

### 핵심 원칙
* 문제는 DB에 저장하지 않음 (서버 코드에서 생성)
* 항상 동일 입력 → 동일 문제 (Deterministic)
* `seed = hash(studentId + classCode + progressIndex)`
* `generateQuestion(progressIndex, seed)`

| fill_blank | 빈칸 채우기 |

| transformation | 문장 변환 |
| error_detection | 틀린 문장 찾기 |

### 레벨별 유형 비율 (%)

| word_order | 35 | 25 | 15 |

| word_meaning | - | - | - |

* **타입 결정 로직**: `type = getTypeByIndex(progressIndex)`
* 동일 유형 3회 연속 금지

### 🧱 템플릿 및 단어 풀 시스템
* 레벨별 단어 제한: beginner(기본), intermediate(과거형 포함), advanced(확장/구문)
* 오답 생성 규칙: 최소 2가지 오류 혼합 (시제, 수일치, 어순, 문법 누락, 유사 단어 등)

### 8.3 🚨 문제 생성 품질 보장 명세 (AI 전달용)

#### 0. 🎯 목표
* 모든 문제는 **정답이 정확히 1개 존재해야 한다**
* 문법적으로 **완전히 올바른 영어 문장만 정답으로 허용**
* 오답은 **그럴듯하지만 명확히 틀린 문장**이어야 한다

#### 1. ❗ 절대 규칙 (Mandatory Rules)
##### 1.1 정답 존재성
* 선택지 4개 중:
  * 정답: 1개
  * 오답: 3개
* 정답이 없거나 2개 이상이면 **문제 폐기**

##### 1.2 문법 검증
정답은 반드시 아래 조건을 모두 만족해야 한다:
* 주어-동사 수일치
* 시제 일관성
* 전치사 사용 정확성
* 자연스러운 의미

##### 1.3 입력 단어 준수
* 반드시 **주어진 단어만 사용**
* 필요 시 문법 변형 허용 (see → sees)
* **임의 단어 추가 금지**

#### 2. 🚫 금지 규칙 (Critical Errors)
##### 2.1 전치사 남용 금지
* 동사별 전치사 사용 여부를 반드시 검증 (예: see -> prep 없음, look -> at 필요)
* 허용되지 않은 전치사 추가 시 → 오답 처리

##### 2.2 존재하지 않는 문장 생성 금지
* 조동사 + 동사 형태 검증 필수 (예: She is sees ❌)

##### 2.3 의미 없는 문장 금지
* 문장이 문법적으로 맞더라도 **의미가 부자연스러우면 오답 처리**

##### 2.4 정답 누락 금지
* 모든 선택지가 틀린 경우 → 문제 폐기

#### 3. 🧠 오답 생성 규칙 (Controlled Errors)
오답은 아래 오류 유형만 사용:
* **수일치 오류**: She see the room ❌
* **시제 오류**: She saw the room ❌ (현재형 문제일 경우)
* **어순 오류**: She the room sees ❌
* **관사/구조 오류**: She sees room ❌

##### ❗ 금지된 오답 방식
* 랜덤 전치사 추가 ❌
* 의미 없는 단어 조합 ❌
* 문장 성립 불가 구조 ❌

#### 4. 🧩 문제 생성 검증 단계 (필수)
문제 생성 후 반드시 실행:
1. **정답 생성**: She sees the room ⭕
2. **오답 생성**: 3개 생성 (서로 다른 오류 유형 사용)
3. **전체 검증**: 정답 1개 여부, 문법 유효성, 오답의 의도성 확인

#### 5. 🔥 추가 위험 케이스 방지
* **동사 유형 혼동**: 자동사 vs 타동사 구분 필수 (go -> 자동사, eat -> 타동사)
* **복수/단수 혼동**: They eats ❌, He eat ❌
* **시제 불일치**: She is went ❌
* **중복 정답 방지**: 내용이 같은 정답 2개 존재 시 즉시 폐기

#### 6. 📏 난이도별 추가 규칙
* **초급**: 단순 문장만 허용, 복합 시제 금지
* **중급**: 과거형 포함 가능, 문장 확장 허용
* **고급**: 복합문 허용, 수능 스타일 문장, 의미 기반 문제 포함

#### 7. 🔒 최종 검증 규칙 (가장 중요)
```js
validate(question) {
  return (
    exactlyOneCorrectAnswer &&
    allChoicesValidSentences &&
    noRandomPrepositions &&
    grammarRulesSatisfied
  )
}
```

### 8.4 🚨 문제 생성 정확도 개선 명세 (동사-목적어 규칙 포함)

#### 0. 🎯 목표
* 모든 문제는 **자연스럽고 실제로 사용 가능한 문장**이어야 한다
* 단순 문법 맞춤이 아니라 **의미까지 자연스러워야 한다**
* 잘못된 단어 조합으로 인해 **정답 자체가 이상한 문제 생성 금지**

#### 1. ❗ 핵심 규칙 (반드시 적용)
##### 1.1 동사-목적어 호환성 필수
문제 생성 시 반드시 동사와 목적어의 호환성을 검증하고, 유효하지 않은 조합인 경우 무조건 재생성한다.

##### 1.2 단어는 “랜덤 조합 금지”
`[She, make, park]`와 같이 자연스러운 문장 생성이 불가능한 조합은 피한다.

##### 1.3 템플릿 기반 생성만 허용
반드시 미리 정의된 문장 구조와 해당 구조에 유효한 목적어 리스트를 사용하여 문제를 생성한다.

#### 2. 🧠 동사 규칙 테이블 (예시)
서버 코드에 아래와 같은 구조의 규칙을 정의하여 사용한다:
* **make**: `{subject} makes {object}` (validObjects: ["cake", "food", "plan", "decision", "homework"])
* **eat**: `{subject} eats {food}` (validObjects: ["apple", "pizza", "rice", "bread"])
* **play**: `{subject} plays {game}` (validObjects: ["soccer", "basketball", "game", "piano"])
* **see**: `{subject} sees {object}` (validObjects: ["movie", "dog", "friend", "teacher"])
* **go**: `{subject} goes to {place}` (validObjects: ["school", "home", "park", "store"])

#### 3. 🚫 금지 케이스
* **잘못된 조합**: `make + park`, `eat + homework`, `go + pizza` 등
* **의미 이상 문장**: `She makes park`, `She eats a car` 등

#### 4. 🧩 문제 생성 흐름 (수정 버전)
1. **Verb 선택**
2. **VerbRules 불러오기**
3. **validObjects 중 하나 선택**
4. **Template 기반 정답 생성**
5. **오답 생성 (엄격히 제한된 오류만 사용)**
6. **전체 검증 (실패 시 재생성)**

#### 5. ❌ 오답 생성 규칙 (엄격 제한)
오답은 아래 유형만 허용:
* **수일치 오류**: She make a cake ❌
* **시제 오류**: She made a cake ❌ (현재형 문제일 경우)
* **구조 오류**: She is make a cake ❌
* **관사 누락**: She makes cake ❌

#### 6. 🔍 최종 검증 로직
```js
validate(question) {
  return (
    exactlyOneCorrectAnswer &&
    allChoicesAreValidStructures &&
    verbObjectCombinationValid &&
    noSemanticErrors
  )
}
```

### 8.5 🚨 애매한 정답(복수 정답) 방지 명세

---

#### 0. 🎯 목표
* 모든 문제는 **정답이 1개만 존재해야 한다**
* 문법적으로 가능하지만 **애매한 선택지(부분 정답)는 절대 허용하지 않는다**
* 학습자가 **혼동할 여지 자체를 제거**한다

---

#### 1. ❗ 핵심 문제 정의
다음과 같은 경우는 **문제 실패로 간주**
* `My mom studies history ⭕ (정답)`
* `My mom studies the history ⚠️ (문법적으로 가능)`
👉 학습자 입장에서 **둘 다 맞는 것처럼 보임 → 문제 폐기 대상**

---

#### 2. 🚫 금지 규칙 (Critical)
##### 2.1 “문법적으로 가능하지만 덜 자연스러운 문장” 금지
* 관사(the, a) 유무에 따라 의미만 달라지는 경우
* 특정 상황에서만 맞는 문장
* 문맥 없이는 판단 불가능한 문장 (예: `She opened the door` - 문맥 필요)

##### 2.2 불가산 명사 + 관사 혼용 금지
* 대상: `history, water, information, advice, homework` 등
* **정답**: `She studies history ⭕`
* **오답**: `She studies the history ❌` (생성 금지 - 혼동 유발)

##### 2.3 복수 정답 가능성 즉시 차단
* `She is running` vs `She runs`가 동시에 선택지로 등장하는 것 금지

---

#### 3. 🧩 안전한 문제 생성 규칙
##### 3.1 “명확하게 틀린 오답만 허용”
오답은 아래 조건 중 하나만 만족해야 한다:
* 문법 오류 (수일치, 시제)
* 구조 오류
* 명백한 형태 오류 (관사 누락 등)

##### 3.2 위험 패턴 회피
* **관사 혼동**: history / the history ❌
* **의미 의존**: context 없이 판단 불가한 문장 ❌

---

#### 4. 🔍 최종 검증 로직
```js
validate(question) {
  return (
    exactlyOneStrictlyCorrectAnswer &&
    noAmbiguousChoices &&
    allChoicesClearlyRightOrWrong
  )
}
```

### 13. 문제 데이터 구조

각 문제 객체는 아래 필드를 필수로 포함합니다:

```typescript
interface Problem {
  id: string;                       // 고유 ID (HMAC 기반)
  level: number;                    // 1~100
  difficulty: 'easy' | 'medium' | 'hard';  // 선택한 난이도
  passage: string;                  // 지문 (고급일수록 길고 복잡)
  question: string;                 // 문제 stem
  choices: string[];                // 4개 선택지
  answer: number;                   // 정답 인덱스 (0~3)
  explanation: string;              // 1~2줄 핵심 해설 (MZ 스타일)
  questionType: 
    | 'vocab' | 'grammar' | 'fact' 
    | 'inference' | 'mainIdea' | 'purpose' 
    | 'attitude' | 'blank' | 'order' | 'comparison';
  wordCount: number;                // 지문 단어 수
}
```

### 14. 시작 레벨 선택 모달 vs 난이도 버튼 관계 (Option A 채택)

#### 14.1. 시작 레벨 선택 모달 (Start Level Modal)
- **표시 조건**: `progressIndex === 0` (신규 학생)일 때 최초 표시.
- **기존 학생**: `progressIndex >= 1`인 경우 모달 없이 기존 `currentLevel`로 즉시 시작.
- **재호출**: 학습 화면 상단의 "레벨 변경" 버튼으로 언제든 다시 열 수 있음.
- **역할**: 학생의 기본 진행 지점인 `currentLevel`을 설정.

#### 14.2. 난이도 버튼 (Easy/Medium/Hard) 제한 및 페널티
- **동작 원리**: 학생이 선택한 난이도 버튼은 **현재 `currentLevel`**을 기준으로 문제 풀의 범위를 결정함.
- **난이도별 출제 범위 (Mock 로직)**:
  - **초급(Easy) 버튼**: `currentLevel - 10` ~ `currentLevel + 5` 구간 문제 (쉬운 유형 위주).
  - **중급(Medium) 버튼**: `currentLevel - 5` ~ `currentLevel + 10` 구간 문제 (적정 유형 위주).
  - **고급(Hard) 버튼**: `currentLevel` ~ `currentLevel + 20` 구간 문제 (심화/수능 유형 위주).
- **포인트 페널티 (Penalty)**:
  - 현재 레벨보다 낮은 난이도의 문제를 풀 경우 보상이 감소함.
  - `-3레벨` 이하 낮은 문제: 획득 포인트 50% (5P).
  - `-6레벨` 이상 낮은 문제: 획득 포인트 0~30% (0P~3P).
- **UI/UX 가이드**:
  - 난이도 버튼 위에 "현재 레벨 기준 추천" 배지 표시.
  - 너무 낮은 난이도 선택 시 경고 토스트: "이 난이도는 포인트가 적게 나와요. 더 도전적인 문제를 풀어보세요!"
  - 현재 레벨(`currentLevel`)을 상단 패널에 항상 크게 노출 (예: "현재 Lv.45").

# 1. ⚙️ 인증 구조

## 공통

👉 **하루상자 로그인 시스템 사용**

---

## 1.1 교사

- 카카오 로그인 (하루상자)
- JWT 그대로 사용
```
{
  "userId":"...",
  "type":"teacher"
}
```

---

## 1.2 학생

- 회원가입 없음 (간편 로그인 구조)
- 최초 로그인 시 인증 후 JWT 발급

👉 인증 방식
1. classCode + nickname + password 검증
2. bcrypt 비교
3. JWT 발급

```
JWT payload
req.user = {
  studentId,
  classCode
}
```

👉 이후 모든 요청은 JWT 사용
```
Authorization: Bearer {token}
```

---

## 1.3 서버 역할

- teacher → JWT 검증
- student → JWT 검증 (로그인 시에만 bcrypt 사용)
